# Git Hub 運用ガイド

## 開発作業の運用手順

| #   | 作業          | 作業内容                                                         |
| :-- | :------------ | :--------------------------------------------------------------- |
| 1   | 課題起票      | GitHubでIssueを立てる                                            |
| 2   | ブランチ作成  | vscodeから課題用ブランチを作成する                               |
| 3   | 作業&コミット | 作業ごとに変更をローカルに記録する                               |
| 4   | プッシュ      | 変更内容をGitHub(リモート)に反映する                             |
| 5   | プルリク作成  | 各課題の作業完了ごとにプルリクを作成する                         |
| 6   | マージ        | 変更差分と実装内容に誤りがないことを確認してからマージを承認する |
| 7   | ブランチ削除  | GitHub(リモート)とローカルブランチを削除                         |

## 1.課題起票

課題を起票すると<mark>課題ID</mark>が確定する。

### ■ 課題名の命名規則

<mark>prefix</mark> : 日本語で課題概要を一言

例：

```
feature:○○一覧画面作成
```

### ■ 使いやすいprefix

| #   | prefix  | 意味                                 |
| :-- | :------ | :----------------------------------- |
| 1   | feature | 新規機能追加                         |
| 2   | fix     | バグ修正                             |
| 3   | docs    | ドキュメント修正（コード修正なし）   |
| 4   | chore   | 雑作業（ビルドやライブラリ追加など） |

### ■ 課題起票に記載する内容（一例）

- 概要
- 仕様(変更設計書)
- 画面イメージ（frontendの場合）
- タスクリスト・修正内容
- 留意事項
- 完了条件

```
何を修正・作成する課題なのか明確にして記載する。
作成時点で曖昧な場合は、徐々に追記しながら記載を育てていく。
```

## 2.ブランチ作成

課題起票に使用した`prefix`と`課題ID`を用いて作業用ブランチを作成する。

### ■ ブランチの命名規則

<mark>prefix</mark>/<mark>#課題ID</mark> 英語で課題概要を一言

例：

```
feature/#1-frontend-post-index
```

### ■ 使いやすい概要説明のキーワード

| #   | キーワード     | 意味     |
| :-- | :------------- | :------- |
| 1   | frontend       | 画面作成 |
| 2   | backend        | 機能作成 |
| 3   | change         | 変更作業 |
| 4   | add            | 追加作業 |
| 5   | bug または fix | バグ修正 |

```
ブランチ名に課題IDを忘れないこと。
2~3個キーワードを並べると、ブランチ名から何の修正なのかイメージしやすくなります。
英語のネーミングに困ったら「ネーミング辞典」で検索してみると参考になります。
```

## 3.作業&コミット

- 作業ごとのコミット(記録)を行う。
- 1日の作業の終わりには必ずコミットする習慣をつける。

### ■ コミットメッセージの命名規則

<mark>prefix</mark>`:日本語で修正内容` <mark>#課題ID</mark>

- prefixをつけることでどんな修正か直感的にわかるようになる。
- 課題IDをつけることで、マージ後もどの課題で修正したのかコミット履歴から遡れる。

### ■ 使いやすいprefix（省略表記でコンパクトに！）

| #   | prefix           | 意味                                         |
| :-- | :--------------- | :------------------------------------------- |
| 1   | feat             | 新規機能追加                                 |
| 2   | fix              | バグ修正                                     |
| 3   | docs             | ドキュメント修正（コード修正なし）           |
| 4   | chore            | 雑作業（ビルドやライブラリ追加など）         |
| 5   | refact           | リファクタリング（機能は変えないコード修正） |
| 6   | perf             | パフォーマンス向上目的の修正                 |
| 7   | style            | レイアウト修正(cssなど)                      |
| 8   | test             | テストコード作成                             |
| 9   | <mark>wip</mark> | 作業途中(Work In Progress)                   |

## 4.プッシュ

```
コミットは、途中断面をローカル上で記録をしている。
リモート環境に反映しないと、状況をチーム共有することはできない。
"wip"を用いながら、途中断面もチーム共有していく。
```

### ■ プッシュができない時

コミット前の修正がある場合、プッシュができないため、`スタッシュ`（退避）を活用する。

## 5.プルリク作成

該当課題のブランチでの修正が完了したらプルリクエスト（略称：プルリク or PR）を行う。

### ■ プルリクタイトルの命名規則

<mark>prefix</mark>/<mark>#課題ID</mark> 日本語で課題概要を一言

- 【prefix】/【#課題ID】はデフォルトのままでok（ブランチ名で使用しているもの）
- 【日本語で課題概要を一言】は課題名と同じでOK

**上記の要領でプルリクタイトルを決めることで、システムにも人間にも理解しやすい構成になる。**

### ■プルリクエストに記載する内容（一例）

- 概要
- 該当課題（ <mark>closes #課題ID</mark> ）
- 実装（修正）内容
- 画面（修正前・修正後）
- 確認方法（確認者にどのような確認をしてほしいのか具体的に記載する）
- 留意事項

```
複数の画面や内容に及ぶ修正は避け、課題に対する修正だけにするのがベター。（多少はOK）
修正内容が多すぎると、変更差分を読んでもそれが正しいのか判断が難しくなる。
また、何をどのように確認してほしいのか記載すると、確認者の確認コストを抑える効果もある。
```

```
※ 【closes #課題ID】
課題とプルリクを紐づけ、プルリクエストがマージされるとリモートの該当課題のクローズも自動化できる便利な機能
```

## 6.マージ

変更差分と実装内容に誤りがないことを確認してから承認する

## 7.ブランチ削除

- ブランチは`リモート`（Github）と`ローカル`（PC）の2か所を削除する必要がある。
- 不要なブランチが多いと、間違えて違うブランチにコミットしてしまったりヒヤリハットにつながる危険性もある。

**定期的な削除習慣があるとベスト**

### ■ Github上のリモートブランチ削除方法

自動削除と手動削除の方法がある

#### ★ リモートブランチの自動削除設定方法 (最初に設定しておくと便利!)

- リポジトリ名の下にある[ Settings ]をクリック
- リポジトリ設定から[Pull Requests]一番下の、[ Automatically delete head branches ]を選択

#### ★ リモートブランチの手動削除設定方法

- プルリクエストがマージされた画面の [ Delete branch ]をクリック

### ■ vscode上のローカルブランチ削除方法

- `main`ブランチを最新の状態にプルする
    ```
    git checkout main
    git pull origin main
    ```
- マージ済みのローカルブランチを削除
    ```
    git Branch -b [ ブランチ名 ]
    ```

#### ★ ブランチ名を調べる方法

- ローカルブランチのみ
    ```
    git branch
    ```
- リモートブランチ含めたすべてのブランチ
    ```
    git branch -a
    ```

### ■ vscode上のリモートブランチ削除方法

**これを忘れると、ローカル上から既に削除されたブランチを選択してしまったりヒヤリハットにつながる**

```
git fetch --prune
```
